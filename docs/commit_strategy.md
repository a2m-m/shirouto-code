# コミット戦略と規約

このドキュメントは、プロジェクトにおけるコミット戦略、コミットメッセージの規約、および巨大な差分（diff）をもつコミットに対する警告の仕組みを定義します。
(`01_OS Overview.md` の §11に基づく、壊れない設計としての規約です)

---

## 1. コミットの基本原則

- **1コミット = 1意図**: ひとつのコミットには、ひとつの論理的な変更（機能追加、バグ修正、リファクタリングなど）だけを含めること。
- **自己完結性**: 可能な限り、そのコミット単体でビルドやテストが通り、論理的に完結している状態を保つこと。
- **巨大コミットの回避**: レビューの負荷を下げ、バグの混入を防ぐため、一度に大量のファイルや行数を変更するコミットは避けること。

---

## 2. コミットメッセージ規約

プロジェクトでは **Conventional Commits** ベースの規約を採用します。
コミットメッセージは以下のフォーマットで、**日本語**で記述してください（`type` などの一部の識別子や固有名詞は除く）。

### フォーマット

```text
<type>: <日本語の説明>

または

<type>(<scope>): <日本語の説明>
```

### Type一覧

| Type | 用途・意味 |
|---|---|
| `feat` | 新しい機能の追加 |
| `fix` | バグの修正 |
| `docs` | ドキュメントのみの変更 |
| `style` | コードの意味に影響を与えない変更（空白、フォーマット、セミコロンの欠落など） |
| `refactor` | バグ修正や機能追加ではない、コードの構造や品質の改善 |
| `perf` | パフォーマンスを向上させるコードの変更 |
| `test` | 不足しているテストの追加、既存テストの修正 |
| `build` | ビルドシステムや外部依存関係に影響する変更 |
| `ci` | CI設定ファイルやスクリプトの変更 |
| `chore` | その他、ソースコードやテストコードを変更しない補助的なツールの変更など |
| `revert` | 以前のコミットの取り消し |

### コミットメッセージの例

- `feat: ユーザーログイン用APIエンドポイントを追加`
- `fix(auth): トークン有効期限の検証漏れを修正`
- `docs: 新しいコミット規約をREADMEに追記`
- `refactor: データベース接続ロジックを別モジュールに分離`

## 3. 巨大diff警告の仕組み

1コミット=1意図の原則を補完し、PRの巨大化を防ぐため、**ローカルでの `git push` 時 (pre-push hook)** にコミットの差分量（diff行数・変更ファイル数）をチェックし、警告を出す仕組みを導入します。

### 仕様と閾値

閾値はリポジトリルートの `os-template.yml` で定義された `policy` に従います。

- **`policy.max_diff_warning`**: 1回のプッシュに含まれる変更行数の目安（デフォルト: `1200`行）
- **`policy.max_file_warning`**: 1回のプッシュに含まれる変更ファイル数の目安（デフォルト: `30`ファイル）

### 検知タイミング

- **`pre-push` フック**: ローカルからリモートへプッシュする直前に、プッシュ対象となるコミットの変更差分を合算して判定します。

### 出力形式・アクション

現在は **Warn（警告のみ）** の段階として導入します。後日、仕組みが安定した後に Block への移行を検討します。

1. **警告対象ではない場合**:
   何も出力されず、そのままプッシュが成功します。
2. **閾値を超過した場合 (Warn)**:
   ターミナルに以下のような警告メッセージが表示されますが、プッシュ自体は許可されます。
   ```text
   WARN: 変更行数 (${変更行数} 行) が目安 (${max_diff_warning} 行) を超過しています。
   WARN: または、変更ファイル数 (${変更ファイル数}) が目安 (${max_file_warning}) を超過しています。
   WARN: 巨大なPRはレビュー負荷を高めます。可能であればIssueやPRの分割を検討してください。
   ```

### 今後の展望

- 現状は実装済みのフックやスクリプトに対する仕様定義のみです（具体的なスクリプト本体は別途実装予定）。
- プロジェクトの成熟度に応じて、CI ワークフローへの組み込みや、ハードリミット（Block）の導入を検討します。
