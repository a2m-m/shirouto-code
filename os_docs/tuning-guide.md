# OSチューニングガイド（改善ループの運用）

このドキュメントは、Project Instance における**「人を注意するのではなく、仕組みを更新する」**ための具体的な改善プロセスを定義します。

## 1. 基本思想と全体フロー

当OSでは、開発中の失敗やルール違反を「ヒューマンエラー」として処理せず、「OS（仕組み）の欠陥」として扱います。問題が発生した場合は、以下のフローで改善ループを回します。

1. **詰まり（Issue）の発生**：ルール違反、バグの再発、不要なコード変更など。
2. **診断**：なぜその問題が起きたのか（指示が弱い？ゲートがない？等）を分類。
3. **対策の立案**：どのファイル（プロンプト、CI、ドキュメント）をどう直すか決定。
4. **実施（仕組みの更新）**：該当する仕組みのコード/ドキュメントを修正。
5. **確認**：修正内容が以降の開発で機能するかを確認し、必要に応じてTemplateへ還元する。

## 2. 改善ループの運用体制とトリガー

### 運用のトリガー
改善ループは、以下のタイミングで実行されます（`01_OS Overview.md` §9⑤ Maintenance & Grooming 相当）。
- 週次（例：週30分）などの定期的なメンテナンスタスクとして。
- スプリントの区切りや、大きな機能のリリース後。
- 致命的な問題や著しい非効率がチーム内で複数回観測されたタイミング。

### 担当者
- **Claude Code**: QA/Architect として問題を分析して具体的な改善方針をIssueとして起票し、Executor として起票されたIssueに従って仕組みのコードやドキュメントを修正・コミットします。

## 3. 4つの改善パターンと具体的手順

問題の性質に応じて、以下の4パターンのいずれかで対処します。

### パターン1: 「指示が弱い」（期待した通りの実装や動きにならない）
最初の命令やコンテキストが不十分な場合に発生します。

- **対象ファイル**: `.github/ISSUE_TEMPLATE/*.yml`, プロジェクト固有のドキュメントテンプレート, `.ai-context.md` の構造。
- **修正の方向性**:
  - Issueテンプレートに「Acceptance Criteria」や「Commit Plan」の必須項目を追加する。
  - AIに対する前提条件や指示の解像度を上げるための記述を追加する。

### パターン2: 「ゲートが弱い」（防ぐべき問題が後工程に流出する）
チェック機構が甘く、テスト不足やスコープ逸脱が発生した場合です。

- **対象ファイル**: `.github/workflows/*.yml` (CI), `os_scripts/run` (pre-push/post-push hooks)。
- **修正の方向性**:
  - CIに新しいLintルール、静的解析、あるいはテストカバレッジの閾値を追加する。
  - Git hookのスクリプトに、巨大コミットの検知など新たなアサーションを追加する。

### パターン3: 「教育が弱い / うるさい」（レビューのフィードバックが不適切）
LCG等のフィードバックが的外れだったり、情報過多でノイズになっている場合です。

- **対象ファイル**: `os_scripts/lcg` などのフィードバック生成ロジック、あるいはそれらが参照するプロンプト設定ファイル。
- **修正の方向性**:
  - 出力量の制限（例: 最大3点までに絞る）。
  - トリガー条件の調整（例: SeverityがNITのものは報告しない）。
  - ルールやガイドラインの解釈に関するプロンプトの微調整。

### パターン4: 「同じ指摘が3回繰り返される」（根本的な規約の浸透不足）
同じようなコードレビューの指摘やミスがプロジェクト内で頻発する場合です。

- **対象ファイル**: `.ai-instructions.md`, プロジェクトルートの `README.md`, `types/` などの共通型定義ファイル, Guardrail や LCG の設定。
- **修正の方向性**:
  - AI共通指示ファイル (`.ai-instructions.md`) に明示的な「やってはいけないこと」「必ずやること」を追加する。
  - プロジェクトの共通ルールとして型定義やアーキテクチャ規約を明文化し、AIに強制する。

## 4. Templateへの還元プロセス（OSの進化）

「Project Instance（個別プロジェクト）」の運用の中で発見され対応した改善策のうち、汎用的に有益だと判断されたものは、「OS Template Repo（テンプレそのもの）」へ還元する必要があります。

1. **効果の確認**: Project Instance内で仕組みの更新が有効に働いているか確認する。
2. **還元Issueの起票**: OS Template Repo 側に対して、該当のプロンプト追加やCI強化といった改善内容を移植するIssueを起票する。
3. **Templateの更新**: OS Template Repo に反映することで、次に `init` される全ての新規プロジェクトが最初からその恩恵を受けられる状態にする（OS全体の進化）。
